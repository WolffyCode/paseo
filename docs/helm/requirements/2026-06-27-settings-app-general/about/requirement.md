## 1. 目标 & 用户价值

**一句话目标**：让用户在一个「关于」面板里一眼看清**这台设备上的 Helm 是什么版本、走哪个发布频道、要不要更新、所连各主机的版本是否对得上、以及去哪儿找社区**。

**用户为什么需要**：

- **「我这套是不是最新的？」** —— 桌面用户最常问的就是版本与更新。把 App 版本、发布频道、软件更新三件事并排放在「关于」里，用户不用去别处找「检查更新」。
- **「客户端和主机版本对不对得上？」** —— Helm 客户端连多台主机；当客户端比某台主机新（或反之），部分功能会因主机能力缺失而不可用。把**全部已连主机的版本**列出来并在不一致时高亮，用户能立刻定位「是哪台主机该升级」，而不是对着一个失效功能猜。
- **「想 Star / 赞助 / 进社区」** —— 给愿意支持项目的用户一条明确出口；这是 fork 自身的社区入口，必须指向 **Helm**，不能停留在上游 Paseo 的链接。
- **诚实地区分平台能力** —— 发布频道与软件更新是**桌面 App 专属**；在 Web / 手机上它们本就不存在，面板据平台只显该平台真有的项，不画无法点的假按钮。

## 2. 范围

### 做（本 tab 覆盖 5 块）

1. **App 版本（本设备）** —— 只读行。全平台。
2. **发布频道（Release channel）** —— 稳定 / 测试二选一，切换即持久化。**桌面 App 专属**。
3. **软件更新（App updates）** —— 检查更新（手动 + 进入页自动静默）/ 安装（带确认弹窗）/ 全状态文案。**桌面 App 专属**。
4. **已连主机版本对比（Connected hosts）** —— 列**全部**已连主机的版本，不一致高亮；无主机时整块隐藏。全平台。
5. **社区链接** —— Star / Sponsor / Community 三按钮，**三个 URL 全部改指 Helm**。全平台。

### 边界与依赖

- **平台**：全 tab 在桌面 / Web / 手机均可进入；其中**发布频道 + 软件更新两块仅在桌面 App（Electron 桌面端）出现**，其余平台这两块不渲染。
- **依赖**：软件更新依赖桌面 App 的自更新能力（仅桌面具备）；主机版本对比依赖各主机上报的版本信息（在线才有，离线显「离线」、未上报显「—」）。
- **关系**：本 tab 属应用段（设备级 / App 自身），与「当前选中主机」无关。**已连主机对比块特意列全部主机、不跟随主机段顶部的主机切换器**——这是它与主机段最大的区别，须保留。

### 不做（本轮显式排除）

- **不实现、不改任何代码、不碰 daemon**（本轮只到内容明细 + 设计稿，见 inventory §3）。
- **不新增**版本/更新/主机对比/社区之外的「关于」项（无 License、无 Credits、无 Changelog、无设备信息导出——YAGNI，无人提出需求前不加）。
- **不改**软件更新的底层机制、不新增更新频道（只「稳定 / 测试」两档，沿用现役）。
- **不在本块做**诊断 / 日志 / 连接排错（那是应用段「诊断」tab，后续批次）。
- **不做**主机的添加 / 删除 / 重命名（主机管理在主机段 / 主壳，本块只读展示版本）。

## 4. 核心流程（端到端 · 每条有入口有出口、无死胡同）

1. **看版本 / 决定要不要更新（桌面）**：进入设置 → 应用段「关于」→ 进入页时**自动静默检查更新** → 状态行显示「已是最新」或「有可用更新 + Ready to install: vX.Y.Z」→
   - 若已是最新：安装按钮显「Update」且禁用，用户读到「已是最新」即可离开，**出口=知情**；
   - 若有更新：用户点「Update to vX.Y.Z」→ 弹确认框 →【确认】开始安装、按钮转「Installing…」→ 安装完成（由桌面 App 完成更新流程）；【取消】回到可安装态，无副作用，**出口明确**。
2. **手动复查更新（桌面）**：用户随时点「Check」→ 按钮转「Checking…」、期间两按钮禁用 → 结束后状态行刷新为「已是最新 / 有可用更新 / 出错」。**出错**则状态/错误行给出原因，用户可再点「Check」重试，**不卡死**。
3. **切发布频道（桌面）**：在「发布频道」点 Stable / Beta → **即时持久化** → 后续「检查更新」按所选频道走。来回切换随时生效、无保存步骤，**出口=已保存**。
4. **核对主机版本（全平台）**：进入「关于」→「已连主机」逐行看每台主机版本 → 看到某行高亮 + 「版本与本设备不一致」即定位到需升级的主机 → 用户据此去主机段 / 主壳处理该主机（**本块只读、不在此操作**，出口=知情并转去对应入口）。**无已连主机**时整块隐藏，用户看到的是「没有这一块」，**不是空白卡片**，无死胡同。
5. **去社区（全平台）**：点 Star / Sponsor / Community → 外部浏览器打开对应 **Helm** 页面 → 回到 App，**出口明确**。

> 平台分叉：在 **Web / 手机**进入「关于」，**发布频道 + 软件更新两块不出现**，用户看到的是「版本 + 已连主机 + 社区」三块——这是该平台的**完整正常态**，不是缺失、不报错。

## 5. 全交互态枚举（逐屏 · 引用 ui.html 锚点）

> 与 UI 设计师同步：下列每个状态都要在 `about/ui.html` 有对应屏/态。锚点名供 UI 对齐，最终以 ui.html 逐屏注解为准。

### `ui.html#s-about`（关于 tab · 桌面默认态）
- 五块自上而下：App 版本 → 发布频道 → 软件更新 →（隔开）已连主机 → 社区链接。
- 默认：版本行只读；频道分段控件高亮当前频道；更新状态行=进入页静默检查的结果；主机块逐行列出；社区三按钮常态。
- hover：频道分段、两个更新按钮、三个社区按钮各有 hover 反馈；只读行无 hover 交互。
- 点击效果：频道分段→切换并持久化；Check→进入检查中；Update→弹确认框；社区按钮→外部打开。

### `ui.html#s-about-mobile`（关于 tab · 非桌面平台）
- 仅三块：App 版本 → 已连主机 → 社区链接。**无发布频道、无软件更新**。此为该平台完整正常态。

### 软件更新·五状态（桌面）
- `ui.html#s-about-update-checking`：状态行「检查中」；Check 显「Checking…」；两按钮禁用。
- `ui.html#s-about-update-uptodate`：状态行「已是最新」；安装按钮显「Update」且禁用。
- `ui.html#s-about-update-ready`：状态行「有可用更新」+「Ready to install: vX.Y.Z」；安装按钮显「Update to vX.Y.Z」可点。
- `ui.html#s-about-update-installing`：安装按钮显「Installing…」；两按钮禁用。
- `ui.html#s-about-update-error`：状态/错误行显示错误信息；Check 可再点重试。
- `ui.html#s-about-update-confirm`：安装确认弹窗（标题 / 正文「将更新这台电脑上的 Helm」/ 安装更新 / 取消）。

### 已连主机·状态（全平台）
- `ui.html#s-about-hosts-online`：在线行显版本；**一致行**普通色；**不一致行**版本值高亮 + 「版本与本设备不一致」。
- `ui.html#s-about-hosts-offline`：离线行显「Offline」；在线未上报版本行显「—」。
- `ui.html#s-about-hosts-empty`：**无任何已连主机 → 整块（含段标题）隐藏**。

### 社区链接·状态（全平台）
- `ui.html#s-about-community`：三按钮默认 / hover；点击外部打开 **Helm** 对应页（URL 待定见 §6）。

## 6. 开放项（送董事长闸-1 定夺）

1. **★ 社区三链接的最终 Helm URL**：Star（Helm 官方代码仓库）/ Sponsor（Helm 赞助页）/ Community（Helm Discord）三者的**正式地址未定**。现状三者均为上游 Paseo 旧址，**必须全改 Helm**。请闸-1 给定三条正式 URL；未定前设计稿与文案先标「指向 Helm 官方仓库 / 赞助 / 社区，最终 URL 待定」。
2. **★ 软件更新状态行的精确文案**：进入页静默检查后，「已是最新 / 有可用更新 / 检查中 / 出错」状态行的**确切用户文案**是否沿用现役？是否需要更口语化的中文表达？（「可安装：vX.Y.Z」「检查中…」「安装中…」等带版本/按钮文案已锁，问的是状态说明行。）
3. **★ 主机版本不一致是否给「去升级」出口**：本块定为**纯只读展示**。是否需要在不一致行上给一个「如何升级这台主机」的提示 / 链接（一句指引，不是按钮操作）？还是保持纯展示、由用户自行去主机段处理？
4. **★ 块 4 标题「已连主机」措辞**：本块实际列「客户端已添加的全部主机」（在线显版本、离线显「离线」），标题用「已连主机 / Connected hosts」是否会让用户误以为只列在线主机？是否改为更准确的措辞（如「主机版本」）？

## 7. 验收标准（逐条可判定 · 能 / 否）

1. 进入应用段「关于」tab，能看到**桌面端 5 块 / 非桌面端 3 块**的正确组合：桌面=版本+频道+更新+主机+社区；Web/手机=版本+主机+社区，无频道、无更新行。
2. **App 版本**行：标题「App version」+ 副文案「This device」+ 右值为本设备版本号；全平台均显示且不可点。
3. **发布频道**（桌面）：分段控件含且仅含「Stable / Beta」两档；点击切换**即时持久化**（无独立保存）；重进页面保持上次选择。
4. **软件更新·自动检查**（桌面）：进入本页**自动发起一次静默检查**，无弹窗打扰；检查结束状态行刷新。
5. **软件更新·按钮态**（桌面）：检查中 Check 显「Checking…」且两按钮禁用；无更新时安装按钮显「Update」且禁用；有更新时显「Update to vX.Y.Z」且可点；安装中显「Installing…」且禁用。
6. **软件更新·确认弹窗**（桌面）：点安装先弹确认框（标题「Install desktop update」/ 正文「This updates Helm on this computer」/ 确认「Install update」/ 取消）；取消不产生任何更新动作。
7. **已连主机·列全部**：本块列出客户端已添加的**全部**主机，**不随主机段顶部主机切换器变化**；切换当前主机后本块所列主机集合不变。
8. **已连主机·每行取值**：在线主机显版本号；离线主机显「Offline」；在线但未上报版本显「—」。
9. **已连主机·不一致高亮**：当客户端与某主机版本不一致时，该行版本值高亮且显示「Version differs from this device」；一致时不高亮、不提示。
10. **已连主机·空态**：没有任何已连主机时，本块（含段标题）**整块不渲染**，不留空卡片。
11. **社区三按钮**：显示 Star / Sponsor / Community 三按钮；点击各自在外部打开对应链接；三链接**指向 Helm**（非 getpaseo/paseo、非 sponsors/boudra、非现状 Discord），最终 URL 以闸-1 定稿为准。
12. **平台诚实**：Web / 手机端不出现任何「发布频道」「软件更新」相关控件，且不因其缺席报错或显示空占位。
