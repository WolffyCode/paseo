> **本需求术语**（过闸后并入 canonical 术语表）：**对决（Agent Race）** = 本功能整体流程；**参赛者（participant）** = 一套「提供方·中转站·模型」三层选型 + 模式，跑同一个 prompt 的一个独立代理；**赛道（track）** = 一个参赛者在并行态里的那一栏；**赢家（winner）** = 用户在对比后选中的那个参赛者。沿用既有术语：Provider（提供方）、Model（模型）、Mode（模式）、Workspace、Worktree、Agent session。「中转站」= 三层选型的中间层（供应商/中转站端点），复用既有自定义供应商能力。

## 1. 目标 & 用户价值

**一句话目标**：给 Helm 加一个**独立的「对决」流程**——把同一个任务一键派给多个参赛者（每个 = 一套 提供方·中转站·模型 + 模式），各自在**隔离 worktree** 并行跑，完成后**并排对比**，选出赢家**一键收尾**（采用赢家、其余归档）。

**用户为什么需要它**：多供应商是 Helm 的核心赌注——「为每个任务选最好的模型」。但今天的主壳只让你**混用**模型、不帮你**选**模型。想知道同一个任务 Claude vs Codex vs Copilot、或同一 provider 不同模型谁做得又快又对，用户得手动开 N 个 worktree、起 N 个代理、再逐个比 diff——极繁琐，于是大家干脆不比，凭感觉选模型。对决把「在真实任务上比模型」变成一键流程，让选型**有依据、可重复、低摩擦**。

**典型场景**：用户拿到一个中等任务（修一个 bug / 写一个函数）。他想知道哪套选型最好 → 开对决 → 写一个 prompt → 勾三个参赛者（Claude Sonnet / Codex / Copilot）→ 开始 → 看三条赛道并行跑 → 完成后并排看各自 diff、耗时（、成本）→ 把最好的那个一键采用进主壳继续干，其余丢弃。原本半小时的手工编排，变成几次点击。

## 2. 范围

### 做（本需求）

- **独立的对决端到端流程**：发起（setup）→ 并行赛道（running）→ 成果对比（compare）→ 收尾（finalize）。四态分别对应 `ui.html#rc1`–`#rc4`。
- **发起**：一个**共享 prompt** + 选 **base（项目 + 分支）** + 勾选 **≥2 个参赛者**；每个参赛者 = 一套 提供方·中转站·模型 三层选型 + 模式。少于 2 个有效参赛者不能开始。
- **并行**：每个参赛者在**各自隔离 worktree**（off 所选 base）里跑同一个 prompt；实时看每条赛道的状态 / 当前动作 / 进度 / 耗时 /（可选）成本；顶部整体进度。
- **进入赛道**：可进入任一赛道看完整对话（**交接到主壳**），不影响其它赛道；可回到对决。
- **对比**：并排看每个参赛者的 diff 摘要（文件数 / 增删行）、产出·结论、耗时、（可选）成本；可展开看完整 diff；在成功完成者中**选赢家**。
- **收尾**：采用赢家（赢家 worktree 置为活动 workspace 并进入主壳）+ 其余参赛者归档（确认）；或「都不满意」→ 全部归档 → 重开。
- **全态健壮性**：失败 / 取消 / 降级 / 部分参赛者挂掉 / 中转站不可用 / 离线 / 重载恢复 的明确行为。
- **进行中可恢复**：离开后再回到入口能回到进行中的对决（参赛者是主机侧持久的代理）。
- 桌面 only，浅色主题（accent #20744A）。

### 不做（明确划出去，防蔓延）

- **不是 Codex 式单对话主壳**（已有）。对决不替代主壳，收尾后**交接回主壳**继续。
- **不是三层选型配置页本身**（复用既有选型能力）。对决里每个参赛者只是**选一套既有三层组合**，不在此重做选型 UI。
- **不是设置**。
- **不做多轮 / 淘汰赛 / bracket**：一个 prompt、一轮、N 个参赛者，只此一轮。
- **不做 AI 自动判定赢家**：由用户比、用户选——这正是价值所在。
- **不做对决进行中增删参赛者**：setup 一旦开始即锁定；要变就重开。
- **不做对决「模板 / 预设参赛者组」保存**（YAGNI，后续再议）。
- **不做跨主机对决**：单主机、单项目 base。
- **不做已完成对决的历史档案 / 排行榜浏览**：对决是即时流程，只保证**进行中**可恢复。
- **不在对决内对所有参赛者群发追问**：追问属主壳；进入某赛道后在主壳里继续即可。
- **不改协议、不改 canonical**：本需求只产出本地一对。

## 3. 流程（端到端）

**发起入口**：桌面主壳里一个明确的「对决（Agent Race）」入口（具体落点由 UI 定）。
- 无进行中对决 → 入口进入发起屏 `#rc1`。
- 有进行中对决 → 入口回到该对决的当前态（`#rc2` 运行 / `#rc3` 对比）。

**正常路径**：
1. 进入 `#rc1`：写共享 prompt；选 base（项目 + 分支）；添加参赛者，每个选满 提供方·中转站·模型 + 模式。当 prompt 非空、base 合法、**有效参赛者 ≥2** 时，「开始」可用。
2. 点「开始」→ 系统为**每个参赛者各建一个隔离 worktree**（off 所选 base），把**同一个 prompt** 并行派给各参赛者 → 进入 `#rc2`。
3. `#rc2`：N 条并排赛道，每条显示 参赛者标识（提供方 / 模型）+ worktree 名 + 实时状态（运行 / 完成 / 报错）+ 当前动作·进度 + 耗时 +（可选）成本；顶部整体进度。
4. 全部参赛者到达终态（完成或报错）→「对比」可用 → 进入 `#rc3`。
5. `#rc3`：并排每个参赛者的 diff 摘要、产出·结论、耗时、（可选）成本；可展开看完整 diff；在**成功完成的参赛者**中选「赢家」。
6. 选定赢家 →「采用赢家」→ `#rc4`：赢家 worktree 置为**活动 workspace** 并进入主壳；其余参赛者（及其 worktree）归档（确认）。

**分叉 / 异常 / 降级**：
- **setup 校验不过**（有效参赛者 <2 / prompt 空 / base 非法 / 某参赛者三层未选满 / 某参赛者所选中转站不可用）→「开始」禁用，并指出未满足项；补齐即可开始。
- **setup 取消** → 退出对决流程，回到入口来处（主壳）；**不建任何 worktree**。
- **running 中单个参赛者报错**（provider / 中转站失败、所选模式不支持等）→ 该赛道标红显示原因，**其它赛道继续**，整体进度照常推进；到全部终态仍可进 `#rc3`，报错者在对比里标为失败、不可被选赢家、不阻塞对比。
- **running 中进入某赛道** → 交接到主壳查看 / 继续该参赛者完整对话；对决在后台继续（参赛者是独立 detached 代理）；从主壳可回到对决，赛道状态已更新。
- **running 中取消整场对决** → 确认 → 全部参赛者及其 worktree 归档 → 退出到空态 / 主壳。
- **running 中取消单个参赛者**（可选，针对明显跑飞的那个）→ 确认 → 归档该 worktree、该赛道移除 / 标记；其余继续。
- **compare 中全部失败**（无任何成功完成者）→ **不是死胡同**：每列显失败原因 + 提供「调整重开」（回 `#rc1`，可保留 prompt）/「全部归档」。
- **finalize 选「都不满意」** → 确认 → 全部归档 → 回 `#rc1` 重开（可保留原 prompt 再战）。
- **采用赢家的可选增强**：合并赢家分支回 base / 开 PR —— **能力门**；能力不可用时该选项隐藏或禁用并解释，**采用赢家（进主壳）+ 其余归档 的保底路径始终可用**。
- **离线 / 主机断连**：任意态进入离线态，显示当前对决标识 + 失联；重连后恢复到对决真实态（参赛者在主机侧仍在跑），无需重发。
- **离开 / 重载客户端**：回到入口可恢复进行中的对决，不丢失。

## 4. 屏与全状态

> 逐屏对应 [./ui.html](./ui.html) 的四个锚点。每屏覆盖：默认 / hover / 活跃·选中 / 运行 / 禁用 / 空 / 加载 / 错误 / 离线 / 右键菜单 / 每个可点元素的点击效果。不适用维度写「不适用」。具体视觉与各态像素表现以 ui.html 注解为准，实现需逐态对齐。

### rc1 发起对决（setup） `ui.html#rc1`

**内容**：共享 prompt 输入；base 选择（项目 + 分支）；参赛者列表（每行 = 提供方·中转站·模型 + 模式 + 移除）；添加参赛者；开始 / 取消。

- **默认**：prompt 空占位提示；base 预填当前上下文（若从某 workspace 进入则带入其项目/分支，否则待选）；参赛者列表**预置两行待填**以引导达成最小可比，每行需选满三层 + 模式才算有效。
- **hover**：参赛者行 hover 显行操作（复制 / 移除）；各按钮 hover 态。
- **活跃·选中**：正在编辑的参赛者行高亮；base 下拉 / 任一选型控件展开态。
- **运行**：不适用（尚未开始）。
- **禁用**：「开始」在校验不过时禁用，旁注未满足项（有效参赛者 <2 / prompt 空 / base 非法）。
- **空**：prompt 空、参赛者未选满 = 校验空态（开始禁用）。
- **加载**：拉取可选 provider / model / 中转站 / 分支 列表时的加载态。
- **错误**：所选某中转站不可用 → 该参赛者行标错且不可参赛；拉取选项失败 → 可重试。
- **离线**：主机断连 → 整屏离线态，开始禁用，提示重连。
- **右键菜单（参赛者行，2 项）**：① **复制为新参赛者**（按同三层 + 模式新增一行，便于「同模型不同模式 / 不同中转站」比）；② **移除该参赛者**（删去该行）。
- **点击效果**：三层任一控件 → 打开既有选型；模式 → 切换该参赛者模式；添加参赛者 → 新增一行；移除 → 删行；**开始**（有效时）→ 建各参赛者 worktree 并 fan-out 同一 prompt → 进 `#rc2`；**取消** → 退出对决流程，不建 worktree。

### rc2 赛道实时态（running） `ui.html#rc2`

**内容**：顶部整体进度；N 条并排赛道，每条 = 参赛者标识（提供方 / 模型）+ worktree 名 + 实时状态 + 当前动作·进度 + 耗时 +（可选）成本；进入赛道；取消整场。

- **默认**：全部赛道运行中，实时跳动。
- **hover**：赛道 hover 显赛道操作（进入 / 取消该参赛者）。
- **活跃·选中**：当前聚焦的赛道高亮。
- **运行**：核心态——每条赛道实时刷新状态 / 当前动作 / 耗时；顶部进度按到达终态的参赛者数推进。
- **禁用**：「对比」在未全部到终态前禁用。
- **空**：不适用（进入此屏必有 ≥2 赛道）。
- **加载**：刚进入时 worktree 创建 / 代理初始化阶段，每条赛道的初始化态。
- **错误**：单条赛道报错标红 + 原因，其它继续；若全部失败 → 仍可进 `#rc3`（看失败原因）或直接重开。
- **离线**：整屏离线，赛道冻结显示最后已知态 + 失联提示；重连恢复。
- **右键菜单（赛道，2 项）**：① **进入该赛道（主壳）**；② **取消该参赛者**（归档其 worktree，二次确认）。
- **点击效果**：进入赛道 → 交接主壳；取消该参赛者 → 确认后归档该 worktree、赛道移除/标记；**取消整场** → 确认后全部归档并退出；**对比**（全终态后）→ `#rc3`。

### rc3 成果对比（compare） `ui.html#rc3`

**内容**：并排每参赛者一列：diff 摘要（文件数 / 增删行）、产出·结论、耗时、（可选）成本；可展开看完整 diff；选赢家；采用赢家。

- **默认**：并排展示全部参赛者结果，折叠态显摘要。
- **hover**：列 hover 显列操作（进入赛道 / 展开 diff / 设为赢家）。
- **活跃·选中**：被选为赢家的列高亮。
- **运行**：不适用（已全部终态）。
- **禁用**：「采用赢家」在未选赢家前禁用；失败的参赛者列「设为赢家」禁用。
- **空**：全部失败 → 空成果态：每列显失败原因 +「调整重开 / 全部归档」。
- **加载**：拉取各 worktree diff 摘要时的加载态。
- **错误**：某列 diff 拉取失败 → 该列可重试，不阻塞其它列对比。
- **离线**：整屏离线，显最后已知摘要 + 失联；重连刷新。
- **右键菜单（参赛者列，3 项）**：① **进入该赛道（主壳）**；② **展开/收起 diff**；③ **设为赢家**（仅成功完成的列可用）。
- **点击效果**：展开 → 看完整 diff；进入 → 主壳；设为赢家 → 选中该列、采用赢家可用；**采用赢家** → `#rc4`。

### rc4 收尾 + 状态合集 `ui.html#rc4`

**内容**：选定赢家后的收尾抉择——采用赢家（+ 合并/开 PR 能力门）其余归档（确认）；「都不满意」全部归档重开。本屏并承载 加载 / 空（未发起）/ 部分参赛者报错 / 某中转站不可用（能力门）/ 离线 的状态合集。

- **默认**：显示赢家 +「采用赢家」+（能力门）合并回 base / 开 PR 选项 +「都不满意」。
- **hover**：各按钮 hover 态。
- **活跃·选中**：选中的收尾动作。
- **运行**：收尾执行中（采用 / 归档 / 合并 进行）的过场态。
- **禁用**：合并 / 开 PR 在能力门关闭时禁用并解释；**采用赢家始终可用（保底）**。
- **空（未发起）**：没有进行中对决 → 空态：简介对决 +「发起对决」入口 → `#rc1`。
- **加载**：收尾动作执行中的加载；空态判定「是否有进行中对决」的加载。
- **错误**：采用 / 归档 / 合并 失败 → 报错 + 可重试；归档为软删，重试不致数据半截。
- **离线**：收尾期间断连 → 离线态，重连后回到收尾真实态。
- **右键菜单**：无（收尾以明确按钮 + 确认弹窗为主）。
- **点击效果**：**采用赢家** → 赢家 worktree 置为活动 workspace 并进主壳 + 其余归档（确认）；**合并回 base / 开 PR**（能力门可用时）→ 走既有合并 / PR 流；**都不满意** → 确认 → 全部归档 → 回 `#rc1`（可留 prompt）；**发起对决**（空态）→ `#rc1`。

**确认弹窗（贯穿各屏）**：取消整场 / 采用赢家（归档其余）/ 都不满意（全部归档）/ 取消单个参赛者，均需二次确认（归档虽为软删，仍不可轻易误触）。

## 5. 边界与依赖

> 凡「现有可见能力」之外的，一律以**依赖 / 能力门**表述，不写实现；客户端按能力在/不在决定显隐，不写降级假实现。

- **并行参赛者**：依赖既有「从一个 prompt 并行起多个独立（detached）代理」+「每个代理在新建隔离 worktree 里跑」的能力。复用，不重造。
- **三层选型**：依赖既有 提供方·中转站·模型 三层选型能力。对决不重做选型 UI，每个参赛者复用之。
- **并排对比**：依赖既有「每个 worktree 的改动 diff（含文件数 / 增删行 / 完整 diff）」能力。
- **归档收尾**：依赖既有「worktree 归档（软删 + 连带其内代理 / worktree 清理）」核心能力——其余参赛者归档即清理其 worktree。
- **能力门 ①（赢家合并回 base）**：现有可见能力 = worktree 创建 / 列表 / 归档 + 通用本地 git 合并 + GitHub PR 流。「**一键把赢家 worktree 合并回 base 分支**」作为收尾增强：是否由既有合并原语组合而成、还是需新增主机能力 —— **列为能力门，由架构确认**。**收尾保底语义不依赖它**：采用赢家 = 赢家 worktree 置为活动 workspace + 进主壳 + 其余归档；合并 / 开 PR 仅为可选项，能力不可用时隐藏 / 禁用并解释。
- **能力门 ②（成本 / Token 对比）**：用量 / 成本由参赛者所属 provider 选报（**并非所有 provider 都报**）→ 成本 / Token 为**可选指标**；某参赛者无该数据时该格留空 / 标「不可用」，**绝不阻塞对比或收尾**。
- **能力门 ③（同 prompt 扇出对齐）**：同一 prompt 派给不同 provider 时，模式（plan / default 等）与参数对齐由既有选型 / 创建能力承载；若某 provider 不支持所选模式，该参赛者在 setup 即标不可参赛（校验）或运行时报错（部分失败路径）。
- **能力门（中转站可用性）**：某中转站不可用 → setup 阶段该参赛者标不可参赛（开始校验拦下）、运行阶段该赛道报错（部分失败）；均不阻塞其它参赛者。
- **依赖（对决分组与恢复）**：把 N 个参赛者关联为「一场对决」（可离开再回、可恢复进行中态）—— 是否由既有代理标签 / 关系组合而成、还是需新增分组能力，**列为依赖待架构确认**；不引入第二真相源（不在客户端另存一份与主机重复的对决状态）。
- **平台**：桌面 only（本轮）；浅色主题 accent #20744A。本需求不做手机 / Web 适配。
- **协议**：不修改协议；新主机能力（若有）以能力门表述。
- **与现有功能关系**：对决是**独立流程**，收尾后交接回既有主壳（活动 workspace + 对话）；不改主壳本身、不改三层选型页、不改设置。

## 6. 验收标准

> 逐条「能 / 否」可判定。测试逐条核。

1. 桌面主壳有明确的「对决」入口；无进行中对决时点击进入发起屏（`#rc1`），有进行中对决时回到其当前态（`#rc2` / `#rc3`）。
2. 发起屏含：一个共享 prompt、base（项目 + 分支）选择、参赛者列表（每行可选 提供方·中转站·模型 + 模式）、添加 / 移除参赛者、开始 / 取消。
3. 有效参赛者少于 2 个（或 prompt 空 / base 非法）时「开始」禁用并指出未满足项；补齐后可用。
4. 点「开始」后，每个参赛者各自在一个**隔离 worktree** 里跑**同一个 prompt**，并进入并排赛道屏（`#rc2`）。
5. 赛道屏每条赛道显示 参赛者标识（提供方 / 模型）+ worktree 名 + 实时状态 + 当前动作·进度 + 耗时；顶部有整体进度。
6. 成本 / Token 列：当参赛者 provider 报告用量时显示，未报告时该格留空 / 标不可用且**不阻塞流程**（能力门 ②）。
7. running 中某参赛者报错时，该赛道标错显示原因，其它赛道继续运行，整体进度继续推进。
8. 从赛道屏可进入任一赛道，交接到主壳查看 / 继续该参赛者完整对话；其它赛道不受影响；可从主壳回到对决且状态已更新。
9. running 中可取消整场对决，确认后全部参赛者及其 worktree 归档，并退出对决流程。
10. 全部参赛者到达终态后「对比」可用；未全部终态前「对比」禁用。
11. 对比屏并排显示每参赛者 diff 摘要（文件数 / 增删行）、产出·结论、耗时（成本可选），且可展开看完整 diff。
12. 只能在成功完成的参赛者里选赢家；失败参赛者不可被选为赢家，且其存在不阻塞对比。
13. 选定赢家后可「采用赢家」：赢家 worktree 成为活动 workspace 并进入主壳，其余参赛者（及 worktree）经确认后归档。
14. 「合并赢家回 base / 开 PR」为能力门：能力可用时可点并生效，不可用时隐藏或禁用并解释；**无论该能力在否，采用赢家（进主壳）+ 其余归档 的保底路径始终可用**（能力门 ①）。
15. 「都不满意」可全部归档并回发起屏重开（可保留原 prompt）；全部失败时同样提供「调整重开 / 全部归档」，无死胡同。
16. 任一屏断连进入离线态，显示当前对决标识 + 失联；重连后恢复到对决真实态（参赛者主机侧仍在），无需重发。
17. 离开对决再回（客户端重载 / 重连），能恢复到进行中对决的当前态，不丢失。
18. 取消整场 / 采用赢家（归档其余）/ 都不满意（全部归档）/ 取消单个参赛者 均需二次确认。
19. 每屏交互态（默认 / hover / 活跃 / 运行 / 禁用 / 空 / 加载 / 错误 / 离线 / 右键）与 `ui.html#rc1`–`#rc4` 一致（抽查至少 hover / 活跃 / 加载 / 错误 / 离线 各一处）。
20. 全程桌面态、浅色主题 accent #20744A；本需求不引入手机 / Web 适配，不修改协议与 canonical。
