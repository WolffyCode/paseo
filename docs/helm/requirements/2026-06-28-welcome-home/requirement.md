## 0. 方向二选一（供 gate-1 纠偏）

董事长「优化首次进入的首页 + 套 codePilot」可落到两处，**二选一**：

- **A（主推）· 丰富首页**：升级「中间画布」无对话时的着陆态为 CodePilot 式丰富首页。**理由**：CodePilot 的「首次进入」本身就是这张丰富首页（根路由直接落到对话页，无对话时渲染时段问候 + 双入口卡 + 最近项目 chips，**并非**连主机向导）；董事长描述的「问候 + composer + 最近对话 / 项目 / 快捷入口」**精确对应**它。本需求按 A 推进。
- **B（备选）· 仅换肤 onboarding 品牌欢迎页**：若董事长本意是把 onboarding 那张一次性品牌欢迎页也换成 codePilot 浅色，那是另一条**纯换肤小支**，与 A 不冲突、可另起需求；不在本需求内。

**待 gate-1 拍板的开放项**（各一句，便于纠偏）：
1. 方向 A / B —— 推荐 **A**。
2. 是否引入「个人助手」（不绑定项目的通用对话）作为第二张入口卡 —— 取决于 Helm 是否已有该概念；**默认不引入**（不凭空造功能），待确认。
3. 「最近对话」与左栏对话树并存 —— 推荐**并存但仅取最近几项**（醒目快捷入口，不复制整棵树）。

## 1. 目标 & 用户价值

**一句话目标**：把用户连上主机后落地的「中间画布 · 无活跃对话」着陆态，从简陋占位升级成 CodePilot 式**丰富首页**（套 codePilot 浅色主题），让用户一落地就**被招呼、能一步开干、能一键回到最近的活儿**。

**用户为什么需要**：
- 用户连上主机后第一眼看到的就是中间画布。现状是个简陋空态占位（codepilot-layout 已明确「Composer 等细节标后续」），**没招呼、没抓手**——用户不知道从哪开始。
- 丰富首页给三件用户真正需要的东西：**问候**（这是哪、现在能干啥）、**中央 Composer**（最直接的「开始新对话」入口）、**快捷入口 / 最近项**（回到上次的项目 / 对话，零翻找）。这正是 CodePilot 首次进入的形态，董事长指明照它做。
- 套 codePilot 主题让首页与外壳**一眼是同一个 app**，不割裂。

## 2. 范围

### 做（本需求 = 只填「中间画布 · 无活跃对话」着陆态的内容）

- **时段问候 hero**：品牌图标 + 随时段变化的问候语（早上好 / 下午好 / 晚上好…）+ 一句简短友好提示（可随上下文细化：通用 / 含当前项目名 / 助手语气）。
- **中央 Composer**：丰富首页的**主入口**——开始一段新对话即从这里输入。**仅把外壳既有的 Composer 摆为首页中央主入口**；其内部（附件 / 思考 / 联网 / 模型选择器等）**以外壳既有 Composer 为准，本需求不重定义**。
- **快捷入口**：
  - 「打开项目 / 选择工作目录」入口（在某项目里开始对话）。
  - 「最近项目」「最近对话」短列表 / chips —— 各取**最近的少数几项**，点一下即进入（对话 → 该对话消息流；项目 → 以该项目为上下文准备新对话）。
- **全状态**：首次进入（无最近项 → 引导态）/ 有最近项 / Composer 聚焦态 / hover 态；以及边界态「**无可用 provider / 模型 → 引导去配置**」（只摆引导抓手，**不设计配置本身**）。
- **主题**：整页套 **codePilot**（GitHub 浅色，默认），与 codepilot-layout **同一套主题来源**，**深色一并适配**；不自带独立配色。

### 不做（划清边界，防打架）

- **不是 onboarding 的品牌欢迎 / 连主机**（那是 `2026-06-24-onboarding`，已过 gate-1）。本需求**前提是「已连上主机」**，不碰欢迎页 / 连接中 / 连接方式选择器 / 错误态。
- **不重画外壳**：统一顶栏 / 4 张浮动卡片 / codePilot 主题来源由 `2026-06-27-codepilot-layout` 负责；本需求**复用**外壳与主题，**只填中间画布的着陆态内容**，不动外壳布局与默认显隐。
- **不改普通对话页**（有活跃对话时的消息流本身）。
- **不重定义 Composer 内部**（附件 / 模型选择器等属外壳 Composer）。
- **不设计 provider / 模型的配置界面**（只放「去配置」引导；配置属主机 / 设置相关的另一需求）。
- **不引入新业务概念**（如「个人助手」），除非 gate-1 确认其存在（见 §0 开放项 2）。

## 3. 流程（端到端）

> 同一「中间画布」的两种态切换；每态有入口、有出口，无死胡同。

- **落地丰富首页**：用户连上主机（onboarding 连接成功 / 启动时已连主机的落点）→ 中间画布无活跃对话 → 呈现丰富首页（问候 hero + 中央 Composer + 快捷入口 + 最近项）。
- **开始新对话（主路径）**：在中央 Composer 输入并发出第一条 → 中间画布**切为消息流**（进入普通对话页），离开着陆态。
- **回到最近**：点「最近对话」某项 → 进入该对话的消息流；点「最近项目」/「打开项目」→ 以该项目为上下文准备新对话（随后经 Composer 发出首条进入消息流）。
- **新建 / 清空回落**：用户新建对话（尚未发首条）或当前无任何活跃对话时 → 中间画布回到这张丰富首页。
- **空态（首次进入 · 无最近项）**：无最近项目 / 对话时，最近区呈**引导态**（提示「从打开项目 / 开始新对话起步」），问候 hero 与中央 Composer 仍在，用户照样能开干。
- **边界 · 无可用 provider / 模型**：检测到无可用 provider / 模型时，首页显示一条**去配置的引导抓手**；点它进入配置入口（属另一需求）；配置完成回到首页即恢复正常开干。**本需求只负责「引导」这一步，不负责配置流程。**
- **Composer 聚焦 / 失焦**：聚焦中央 Composer → 进入聚焦态（其余元素让位、突出输入）；失焦未发送 → 回默认态。

## 4. 屏与全状态

> 逐屏对应 [ui.html](./ui.html) 锚点 `ui.html#sX`，与 UI 设计师对齐锚点。桌面优先（紧凑 / 手机态随外壳后续统一适配）。**浅 / 深两套**：每屏默认 GitHub 浅色，深色随 codePilot 自动适配（在各屏注解块标深色变体）。

- **s1 丰富首页 · 默认（有最近项）** `ui.html#s1`：标准落地态——居中**时段问候 hero**（品牌图标 + 问候语 + 一句友好提示）+ 其下**中央 Composer**（首页主入口）+ **快捷入口**（打开项目 / 选择工作目录）+ **最近项目 / 最近对话**短列表。覆盖：默认 / hover（卡片 / chips / 最近项悬浮）/ 各可点元素的点击效果（见 §3 出口）。
- **s2 首次进入 · 空态（无最近项 · 引导态）** `ui.html#s2`：同一 hero + 中央 Composer，但最近区为**空 → 引导态**（「从打开项目 / 开始新对话起步」）。覆盖：空态文案 + 主引导抓手点击效果。
- **s3 Composer 聚焦 + hover / active 态** `ui.html#s3`：中央 Composer 聚焦态（突出输入、其余让位）+ 快捷入口 / 最近项的 hover / active；以及发出首条后离开着陆态的过渡说明（注解，不画进框）。
- **s4 边界 · 无可用 provider / 模型（引导去配置）** `ui.html#s4`：首页叠加一条「去配置」引导抓手的态；覆盖：抓手默认 / hover / 点击效果（进入配置入口，属另一需求）。**配置界面本身不在本屏。**

每屏的完整交互态以 ui.html 逐屏「交互注解块」（布局 / 状态 / 右键菜单 / 点击效果）为准。

## 5. 边界与依赖

- **与 `2026-06-27-codepilot-layout` 的关系（填充，非取代）**：本需求**填充**该外壳里「中间画布 · 对话空态占位」这一被标「后续」的内部细节——把占位升级为丰富首页。**外壳（顶栏 / 4 卡片 / 主题来源 / 默认显隐 / 拖拽）一律不动**。回写 canonical 时，原「对话空态占位」即等于本丰富首页。
- **与 `2026-06-24-onboarding` 的关系（承接落点）**：onboarding「连接成功 → 进入主页空态」的那个**落点就是本丰富首页**；onboarding 明确划走的「provider / 模型配置由主页空态引导」**由本首页承接其『引导』抓手**（配置本体仍属另一需求）。本需求不改 onboarding 任何屏与连接逻辑。
- **与普通对话页的关系**：二者是同一「中间画布」的两种态（无对话 = 本首页；有对话 = 消息流）；本需求只管前者，不改后者。
- **主题依赖**：依赖 codepilot-layout 提供的统一主题来源（codePilot 浅 / 深）；首页不得自带冲突配色，须与外壳一眼同一 app。
- **数据依赖（能力级）**：依赖「能取到最近项目 / 最近对话」「能判定当前有无可用 provider / 模型」这类既有信息——**只读、只展示、只引导**，本需求不新增其采集 / 配置机制。
- **平台**：桌面优先；紧凑 / 手机态随外壳后续统一适配。
- **无老用户 / 迁移**（fork 未上线）。**不改协议 / RPC / `server_info.features.*`**。

## 6. 验收标准（逐条可判定 · 能 / 否）

1. 连上主机后、中间画布无活跃对话时，落地的是这张**丰富首页**（含问候 hero + 中央 Composer + 快捷入口 + 最近项），**取代**原简陋空态占位；它是中间画布的一种着陆态，**不是**新增独立路由 / 页面。
2. 首页存在**时段问候 hero**：品牌图标 + 问候语 + 一句友好提示；问候语**随时段变化**（如早 / 午 / 晚不同）。
3. 首页中央存在 **Composer** 作为「开始新对话」主入口；在其中发出第一条消息后，中间画布**切换为消息流**（离开着陆态）。
4. 本需求**未重定义** Composer 内部（附件 / 模型选择器等仍以外壳既有 Composer 为准）。
5. 首页存在「**打开项目 / 选择工作目录**」快捷入口；点击后以该项目为上下文准备新对话。
6. 首页存在「**最近项目**」「**最近对话**」短列表（各取最近少数几项）；点最近对话 → 进入该对话消息流，点最近项目 → 以该项目为上下文准备新对话。「最近对话」**不复制整棵左栏对话树**，仅取最近几项。
7. **无最近项**时，最近区呈**引导态**（引导从打开项目 / 新对话起步），且 hero 与中央 Composer 仍在、仍可开干。
8. 存在 **Composer 聚焦态**（聚焦时突出输入）与 hover 态（卡片 / chips / 最近项悬浮）。
9. **无可用 provider / 模型**时，首页显示「**去配置**」引导抓手并可点击进入配置入口；本需求**不含**配置界面本身。
10. 整页套 **codePilot**（GitHub 浅色，默认），与 codepilot-layout **同一套主题来源**，**深色一并适配**；首页不自带独立配色，与外壳**一眼同一 app**。
11. 本需求**不改**：外壳（顶栏 / 4 卡片 / 主题来源 / 默认显隐 / 拖拽）、普通对话页消息流、onboarding 任一屏与连接逻辑、provider / 模型配置界面。
12. s1（默认有最近项）/ s2（空态无最近项）/ s3（Composer 聚焦 + hover）/ s4（无 provider 引导配置）四屏在 ui.html 均有对应锚点，各屏交互注解块覆盖 默认 / hover / 活跃 / 空 / 浅深主题及各可点元素点击效果。
13. 文档明确 §0 的**方向二选一**（推荐 A · 丰富首页）与三条开放项，供 gate-1 拍板。
